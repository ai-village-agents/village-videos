## SCENE 01 — Hook
> NARRATION: As large language models grow, we face a fundamental physics crisis: a direct trade-off between capability and computation. Training and running a trillion-parameter dense model requires unfathomable compute power, threatening to melt silicon and bankrupt enterprises. But what if we could scale a model's knowledge capacity to hundreds of billions of parameters while keeping its computational footprint constant? This is the core genius of Mixture of Experts, or MoE. Today, we will explore MoE routing: the high-performance dispatching algorithm that powers modern models like Mixtral and DeepSeek.

## SCENE 02 — The Dense Scaling Trap
> NARRATION: In a traditional dense neural network, every single token must pass through every single weight parameter in the model. If a model has eighty billion parameters, then every word generated requires eighty billion active FLOPs. This is the dense scaling trap. Expanding model capability requires increasing active parameters, which directly raises compute latency, hardware costs, and energy requirements. To break this linear relationship, we must transition from dense compute to sparse compute, activating only the precise neural pathways we need for each specific token.

## SCENE 03 — The Sparse MoE Architecture
> NARRATION: A Sparse Mixture of Experts network replaces the standard dense Feed-Forward layer with a set of parallel subnetworks called "Experts." For instance, a model might have eight, sixteen, or even two hundred and fifty-six experts. Crucially, each individual expert is a standard, compact neural network. When a token enters the MoE layer, it doesn't visit all experts. Instead, it is dynamically routed to a tiny fraction of them—typically just the top-one or top-two experts. By doing this, we keep the total active parameters per token extremely low, even as the total model capacity grows massive.

## SCENE 04 — The Gating Network (The Router)
> NARRATION: At the heart of this sparse architecture is the gating network, or the Router. The Router is a simple, lightweight linear layer that takes a token's hidden representation and decides which experts are best equipped to process it. It computes a numerical routing score for each expert by performing a matrix multiplication between the token's representation and a learned routing weight matrix. This produces a vector of raw logits—one for each expert—representing the token's affinity for that specific specialist.

## SCENE 05 — Softmax Top-K Selection
> NARRATION: To enforce sparsity, we apply a Top-K selection on the routing logits. If we are using a Top-Two gating strategy, we identify the two experts with the highest raw routing scores, set the scores of all other experts to negative infinity, and apply the Softmax function over the remaining values. This produces a sparse gating vector where only the two chosen experts have non-zero probabilities, while the rest are zero. This gating vector acts as a set of dynamic blending weights, determining how much influence each selected expert has on the final output.

## SCENE 06 — The Routing Collapse Failure Mode
> NARRATION: But training such a routing mechanism is notoriously unstable. Left to its own devices, a router is prone to a catastrophic failure mode known as routing collapse. Early in training, a single expert might perform slightly better than the others. The router, seeking to minimize loss, begins sending more tokens to this expert. This expert gets more gradient updates, making it even better, which causes the router to send even more tokens. Soon, all tokens are routed to a single expert, reducing the model back to a small, under-capacity dense model while leaving other experts completely untrained and idle.

## SCENE 07 — The Load-Balancing Loss
> NARRATION: To prevent routing collapse, researchers introduce an auxiliary load-balancing loss during training. This loss functions as a mathematical penalty for uneven token distribution. It is calculated by taking the dot product of two vectors: the fraction of tokens actually routed to each expert in a batch, and the average routing probability assigned to each expert across that same batch. By minimizing this load-balancing loss, we force the router to distribute tokens evenly, ensuring that all experts are utilized and trained to their full capacity.

## SCENE 08 — Expert Capacity Limits and Padding
> NARRATION: In hardware implementations, dynamic routing presents a massive engineering challenge: GPU parallelization requires static tensor shapes. If different experts receive vastly different numbers of tokens, we cannot execute them efficiently in parallel. To solve this, models define an Expert Capacity limit—the maximum number of tokens a single expert can process in a forward pass. If too few tokens are routed to an expert, its input tensor is padded with zeros. If too many tokens are sent, the excess tokens are dropped entirely, bypassing the expert via a residual connection. This maintains stable GPU execution times.

## SCENE 09 — The Efficiency Frontier
> NARRATION: The results of this engineering masterstroke are revolutionary. Take Mixtral eight-by-seven-billion, for example. It possesses forty-seven billion total parameters, but activates only thirteen billion parameters per token. This allows it to deliver the latency and speed of a thirteen-billion parameter model while matching or exceeding the accuracy and quality of a forty-seven-billion parameter dense giant. By separating total parameter capacity from active computational cost, Mixture of Experts pushes the boundaries of what is possible in modern AI.
